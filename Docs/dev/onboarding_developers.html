Guia de Iniciação para Developers — RideReady

Bem-vind@ ao projeto. Este guia leva-te de um clone fresco até à aplicação a correr, e poupa-te os tropeções que já custaram horas a quem cá esteve antes. Lê isto antes de escrever código.

Documentos irmãos: README.md (visão geral) · database-migrations.md (migrações em detalhe) · Docs/ (especificação consolidada, planeamento, decisões).

1. Pré-requisitos

O quê Notas
.NET 8 SDK dotnet --version deve devolver 8.x
SQL Server Instância local ou LocalDB (vem com o Visual Studio)
Ferramentas EF Core dotnet tool install --global dotnet-ef
Servidor SMTP de desenvolvimento Obrigatório — ver §4. smtp4dev: dotnet tool install -g Rnwood.Smtp4dev
Visual Studio 2022+ (ou Rider/VS Code) Solução: RideReady.sln

2. Do clone ao primeiro arranque

git clone https://github.com/joao-luizi/M4_ProjectoFinal.git
cd M4_ProjectoFinal

2.1 Configuração — o appsettings.json real não está no Git (de propósito: o .gitignore exclui appsettings*.json, exceto o example). Cria o teu:

cp RideReady/appsettings.example.json RideReady/appsettings.json

Depois ajusta as connection strings ao teu ambiente — o example traz a variante para SQL Server local ativa e a variante LocalDB em linhas _LocalDB prontas a trocar. ⚠️ Nunca commits o appsettings.json nem credenciais de servidores.

2.2 Bases de dados — são duas (decisão de arquitetura: identidade separada do domínio), por isso são dois comandos:

dotnet ef database update --context AppIdentityDbContext --project RepositoryLibrary --startup-project RideReady
dotnet ef database update --context RideReadyDbContext --project RepositoryLibrary --startup-project RideReady

Detalhes, criação de novas migrações e resolução de problemas: database-migrations.md.

2.3 Arrancar:

dotnet run --project RideReady

→ https://localhost:56116 — atenção ao https: o endereço http dá "página não está a funcionar" e já enganou muita gente nesta equipa.

3. Mapa da solução (60 segundos)

  • RepositoryLibrary/ — domínio + dados. Organizado por vertical slices: cada pasta em Features/ (Lessons, Horses, Users...) contém os seus DTOs, Entities, Interfaces, Repositories e Services. Procura por funcionalidade, não por tipo.
  • RideReady/ — a app Blazor Server. Páginas e componentes em Components/Features/<área>/; componentes reutilizáveis (StatCard, EmptyState, UserAvatar) em Components/Shared/. A identidade visual vive em wwwroot/css/rideready-theme.css (tokens --rr-*) e a camada de animação em wwwroot/js/rideready-motion.js.
  • RideReadyAPI/ — Web API com JWT (Swagger incluído em dev).

4. O email é parte do fluxo — liga o SMTP antes de testar

O registo de contas e a recuperação de password enviam email de confirmação, e o SmtpEmailSender foi desenhado para falhar alto (lança exceção se o envio falhar — decisão deliberada, documentada em Docs/). Tradução prática: sem um servidor SMTP de dev a correr no porto 25, o registo rebenta.

smtp4dev   # e abre a UI dele para veres os emails recebidos

5. Convenções da equipa

  • Branches: feature/<nome> a partir de main atualizado; merge por PR. Trabalho de risco alto isolado em branches próprios.
  • Commits: em inglês, no passado — ex.: Added welcome toast after login, Fixed lessons calendar initial load. Correções de bugs não se misturam com cosmética no mesmo commit.
  • Idioma da UI: PT-PT. Textos novos nascem em português.
  • Cores: usar os tokens --rr-* do tema, nunca hex soltos. O coral (--rr-alert) é exclusivo para estados maus (erros, atrasos, lotado) — não o uses como decoração.
  • Dados sensíveis: nunca no Git (ver §2.1). Imagens de runtime (wwwroot/Images/users/) também não.

6. Armadilhas conhecidas (aprendidas a chorar)

  1. Páginas de Identity ficam SSR. Não apliques @rendermode InteractiveServer a Login/Register/etc. — componentes interativos do Radzen aí causam NullReferenceException. Interatividade nessas páginas faz-se com JS estático (vê o padrão no rideready-motion.js).
  2. RedirectManager.RedirectTo(...) não pára a execução. O código a seguir à chamada continua a correr — põe return explícito depois, ou colhes NullReferenceException (já aconteceu três vezes neste projeto).
  3. CSS scoped novo (.razor.css acabado de criar) exige restart completo da app; alterar um existente basta Ctrl+F5. Esquecer isto = "o meu CSS não funciona".
  4. CSS scoped não entra em componentes filhos — usa ::deep para estilizar markup de componentes Radzen dentro do teu.
  5. O tema do Radzen carrega depois do nosso e ganha empates de especificidade. Para o vencer: variáveis --rz-* em :root:has(.rz-layout) com !important (vê os exemplos no fim do rideready-theme.css).
  6. var() não valida tipos — pintar um fundo com var(--rr-base) (que é 18px) falha em silêncio. Confirma que a variável é do tipo certo.
  7. @ literal em Razor escreve-se @@ ("Bem-vind@@" → "Bem-vind@").
  8. string.GetHashCode() é aleatorizado por execução no .NET moderno — para hashes determinísticos (ex.: cor do UserAvatar), usa um hash à mão (FNV-1a no código, comentado).
  9. Conflitos de merge esquecidos (<<<<<<<) produzem cascatas de erros Razor (RZ1006/RZ9980) em linhas inocentes — o erro aparece longe da causa. Despiste: Ctrl+Shift+F por <<<<<<< na solução inteira.

7. Estado real do produto — lê antes de assumir

Nem tudo o que o código sugere está ligado. Três exemplos que enganam (detalhe completo na §8.2 e §9 da Especificação Consolidada):

  • BookingService.CanBook (bloqueio de marcações por pagamento em atraso) existe mas está comentado;
  • IsAvailable() (regras de carga/descanso dos cavalos) existe mas nunca é chamado na criação de aulas;
  • o envio de email no cancelamento de aulas está comentado (a infraestrutura funciona — confirmação de conta e recuperação de password usam-na).

Os dashboards correm em modo de demonstração assumido (faixa âmbar visível); a receita de religação aos serviços está em comentário em cada um.
